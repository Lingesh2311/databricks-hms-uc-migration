Migration Runbook

Hive Metastore to Unity Catalog: User-Asset Migration

How an existing estate of notebooks, dashboards, and mount references is converted from Hive Metastore to Unity Catalog: discovered, scored, reviewed, patched, and reconciled, one stage at a time.

Operational companion to the migration toolkit (the databricks-hms-uc-migration package). Every step maps to a script you run. v1.0.0.

What this is

This is the operational guide to the toolkit that migrates user assets from Hive Metastore (HMS) to Unity Catalog (UC). It does not migrate tables. It migrates the code and dashboards that reference those tables: the hard-coded hive_metastore table names and /mnt/... mount paths scattered across notebooks and dashboards that break the moment the underlying tables move.

The toolkit gives you five things, one per stage. A discovery job that inventories your assets and mounts. A scanner that parses each asset, finds every HMS reference, proposes the UC replacement, and scores how confident it is. A tiered review workflow so a human approves the changes, spending effort only where it is needed. An apply job that backs up and patches the assets in place. And a validation job that reconciles before and after and records every decision.

Who this is for The engineer running the migration on an estate, and anyone covering it. It assumes a Databricks workspace with Unity Catalog, a SQL warehouse, and the ability to run Python (locally with the CLI profile, or as workspace jobs).

The two halves of UC migration

The single most common point of confusion is what this tool does and does not do. UC migration has two halves, and they are separate jobs.

1. The metastore objects

Moving the tables, grants, and the HMS-to-UC table mapping itself. This is handled by your existing table-migration tooling (for example UCX). This toolkit does not do this.

2. The user assets

The notebooks, dashboards, and scripts that still reference the old tables and mount paths. When the tables move, this code breaks. This is what this toolkit migrates.

The link between the two is the hms_to_uc_mapping table: the answer key that says sales_db.transactions now lives at analytics.sales.transactions. In production you populate it from your table-migration output (for example UCX's table mapping). For a trial you seed it directly. It is read at scan time only. It does not move data.

The control tables

All state lives in four Delta tables in one UC schema (<catalog>.<schema>). You create them once in Step 0.

TableWritten byHolds
hms_to_uc_mappingYou (or your table-migration output)The HMS-to-UC answer key
mount_point_registryStage 1 DiscoveryOne row per mount point, linked to a namespace
demo_asset_inventoryStage 1 (or the seed step)One row per asset in scope
migration_manifestStage 2 Scan, updated by Review, Apply, ValidateOne row per HMS reference found: original, proposal, confidence, status

The migration_manifest is the spine of the whole flow. Every later stage reads and updates it. Its status moves proposed → approved / overridden / skipped → applied / failed.

The flow at a glance

Stage 1: Discover enumerate notebooks, dashboards, mounts → mount_point_registry, demo_asset_inventory
↓
Stage 2: Scan and propose parse each asset, find references, propose UC targets, score confidence → migration_manifest (proposed)
↓
Stage 3: Review human approves, by tier → manifest status + final replacement
↓
Stage 4: Apply back up, patch source, re-import → assets migrated, manifest applied
↓
Stage 5: Validate and record reconcile pre/post, log every decision → mismatches flagged, audit trail

Stages 1, 2, 4, 5 are scripts you run (as workspace jobs or locally). Stage 3 is interactive: notebooks plus a Genie Code skill, driven by a human reviewer.

0One-time setup

Do this once per workspace, before your first migration run.

0.1 Get the toolkit

Clone the toolkit locally for command-line runs, or add it as a workspace Git folder to run the stages as jobs. The runnable code lives under scripts/ (the stage jobs, the scanner, the seed, the review notebooks) with a databricks.yml bundle.

# local
git clone <your_toolkit_repo> databricks-hms-uc-migration
cd databricks-hms-uc-migration
pip install databricks-sdk sqlglot mlflow ipywidgets

In Databricks for the jobs path: Workspace, then your user, then Add, then Git folder, then paste the repository URL.

0.2 Create the schema and control tables

One schema, four tables. Run this SQL once. (DDL is also in references/pipeline-stages.md.)

CREATE SCHEMA IF NOT EXISTS <catalog>.<schema>;

CREATE TABLE IF NOT EXISTS <catalog>.<schema>.hms_to_uc_mapping (
  hms_namespace STRING, hms_table STRING,
  uc_catalog STRING, uc_schema STRING, uc_table STRING,
  mount_path STRING, notes STRING) USING DELTA;

CREATE TABLE IF NOT EXISTS <catalog>.<schema>.mount_point_registry (
  mount_path STRING, adls_target STRING, hms_namespace STRING,
  discovered_at TIMESTAMP) USING DELTA;

CREATE TABLE IF NOT EXISTS <catalog>.<schema>.demo_asset_inventory (
  asset_type STRING, path STRING, name STRING, tier STRING,
  hms_ref_count INT, seeded_at TIMESTAMP) USING DELTA;

CREATE TABLE IF NOT EXISTS <catalog>.<schema>.migration_manifest (
  manifest_id STRING, asset_type STRING, asset_path STRING, cell_index INT,
  original_reference STRING, reference_kind STRING, proposed_replacement STRING,
  confidence DOUBLE, status STRING, final_replacement STRING, reviewer STRING,
  reviewed_at TIMESTAMP, applied_at TIMESTAMP, backup_path STRING,
  scanned_at TIMESTAMP) USING DELTA;

0.3 Pick your run mode

Every stage runs two ways. Choose per your workspace.

ModeHowWhen
As workspace jobsdatabricks bundle deploy -t dev then databricks bundle run hms_discovery -t devProduction runs, scheduling, no local Python
Locally via the CLI profilepython3 scripts/jobs/stage1_discovery.py --profile <profile> ...Iterating, evaluation, a controlled run from a laptop
Parameters every stage takes --catalog, --schema, and where relevant --demo-root, --warehouse-id, --profile. As a job, the same values come from the task parameters. Nothing environment-specific is hardcoded in the scripts. If --warehouse-id is omitted, the first available warehouse is used.

1Seed the mapping

Populate hms_to_uc_mapping with your HMS-to-UC answer key: one row per table you are migrating, and one row per mount point. In production this comes from your table-migration output; below is the manual form.

INSERT INTO <catalog>.<schema>.hms_to_uc_mapping VALUES
 ('sales_db','transactions','analytics','sales','transactions', NULL, 'static table ref'),
 ('finance_db','ledger','analytics','finance','ledger', NULL, 'static table ref'),
 -- a mount row: hms_table NULL, mount_path set
 ('sales_db', NULL, 'analytics','sales', NULL, '/mnt/sales-raw', 'mount resolves to sales_db');
Do not invent UC targets A reference with no mapping entry is scored low and routed to human review, never auto-resolved. Seed the mapping from your real table-migration output rather than guessing target names.

2(Optional) seed a demo estate

To trial the full flow before pointing it at real assets, seed a synthetic estate: notebooks across the three confidence tiers, plus dashboard query specs. The generator is non-PII and deterministic.

python3 scripts/setup/seed_demo_assets.py \
  --profile <profile> --catalog <catalog> --schema <schema> \
  --demo-root /Workspace/Shared/hms_uc_migration_demo/demo_notebooks \
  --warehouse-id <warehouse_id>

Add --local-only to write the generated notebooks to disk without importing them, if you just want to inspect the fixtures first. This step also populates demo_asset_inventory.

3Discover

Inventory the assets in scope and the mount points. This writes mount_point_registry and confirms the asset list.

python3 scripts/jobs/stage1_discovery.py \
  --profile <profile> --catalog <catalog> --schema <schema> \
  --demo-root <workspace_folder_to_scan> --warehouse-id <warehouse_id>

For a real estate, point --demo-root at the workspace folder holding the team's notebooks. Verify the registry filled in:

SELECT mount_path, hms_namespace FROM <catalog>.<schema>.mount_point_registry ORDER BY mount_path;
Mounts on serverless The job reads dbutils.fs.mounts() when available. On serverless that call is not available, so the job falls back to a configured mount set and logs that it did so. The fallback is never silent.

4Scan and propose

Read every inventoried asset, find each HMS reference, propose the UC replacement, and score confidence. This populates the migration_manifest with one proposed row per reference.

python3 scripts/jobs/stage2_scan_propose.py \
  --profile <profile> --catalog <catalog> --schema <schema> \
  --warehouse-id <warehouse_id>

The scanner is pure Python: sqlglot for SQL cells, the standard library ast for Python cells. It exports each notebook's source, finds references, cross-references the mount registry and mapping, and writes the manifest. Verify the spread across confidence tiers before reviewing:

SELECT
  CASE WHEN confidence >= 0.85 THEN 'high'
       WHEN confidence >= 0.45 THEN 'medium'
       ELSE 'needs_judgement' END AS tier,
  reference_kind, count(*) AS n
FROM <catalog>.<schema>.migration_manifest
GROUP BY 1, 2 ORDER BY 1, 2;
Nothing is skipped silently References built at runtime (an f-string mount, a table name assembled from a variable) cannot be resolved statically. The scanner still records them, scored low, so they reach human review. A migration tool that silently skips what it cannot resolve is a tool that quietly breaks code.

5Review and approve

Proposals are routed by confidence so reviewers spend effort only where it is needed. There are three surfaces. Each writes status and final_replacement back to the manifest.

5a. High confidence: batch approve

Open scripts/review/review_simple.py as a notebook. It lists the high-confidence proposals; one cell approves the batch.

UPDATE <catalog>.<schema>.migration_manifest
SET status='approved', final_replacement=proposed_replacement,
    reviewer=current_user(), reviewed_at=current_timestamp()
WHERE confidence >= 0.85 AND status='proposed';

5b. Medium confidence: side-by-side review

Open scripts/review/review_complex.py. It renders an editable card per reference (original vs proposed) with Approve and Override buttons that write the row immediately.

5c. Needs judgement: guided review

For the ambiguous tier, open Genie Code on the row and use the hms-migration-review workspace skill (under scripts/workspace_skill/). It loads the manifest row, shows the source cell in context, enumerates candidate targets from the registry and mapping, and records your decision (overridden, or skipped if it should not migrate). It never sets a target you have not confirmed.

6Apply

Write the approved and overridden changes back into the assets. Every original is backed up to a UC Volume first, so every change is reversible. Only reviewed rows are touched.

python3 scripts/jobs/stage4_apply.py \
  --profile <profile> --catalog <catalog> --schema <schema> \
  --warehouse-id <warehouse_id>

For each affected notebook the job exports the source, backs it up to /Volumes/<catalog>/<schema>/backups/, replaces each original_reference with its final_replacement, re-imports the patched notebook, and stamps the manifest (applied, with applied_at and backup_path). Spot-check a patched asset:

-- before:  spark.read.table("sales_db.transactions")
-- after:   spark.read.table("analytics.sales.transactions")
Reversible by construction Re-import overwrites the notebook, so the original is backed up first. To roll back, restore the file from its backup_path in the Volume.

7Validate and record

A migration that ran is not the same as a migration that is correct. This stage reconciles each migrated table before and after, flags any mismatch back to the manifest, and logs every review decision as an audit record.

python3 scripts/jobs/stage5_validate.py \
  --profile <profile> --catalog <catalog> --schema <schema> \
  --warehouse-id <warehouse_id> \
  --experiment-path /Users/<you>/hms_uc_migration_audit

For each migrated table the job compares the pre-migration source and the UC target on row count and column set. A mismatch sets the related manifest rows to failed. Every reviewed decision is written as a tracking run (one per reference, plus a summary), so you have a queryable record of who approved what, at what confidence, and whether it reconciled. Check what failed:

SELECT asset_path, original_reference, final_replacement
FROM <catalog>.<schema>.migration_manifest WHERE status='failed';

8Iterate: fix here or extend the scanner

Some issues are a one-off edit; some mean the scanner will keep missing the same pattern across the estate. The decision is where to fix.

SituationWhat to fix
One proposal is wrong for a reference unique to that assetOverride it in Step 5 (the review surface). Do not touch the scanner.
A reference pattern is consistently missed or mis-scored across assetsExtend the scanner. Add a failing test first, then the rule.

The scanner is test-first. Add a case to tests/test_scanner.py, watch it fail, then extend scripts/scanner/core.py until green:

pytest tests/
Why test-first matters here The scanner is pure Python with no Databricks dependency, so its rules are fully testable offline. Adding the test first proves the new rule catches what it should before you trust it against a real estate.

9Roll out across the estate

The same flow scales from one team to the whole estate. The recommended approach for a large estate:

  1. Discovery-only first. Run Stages 1 and 2 against one team's folder and read the manifest. You see the worklist and the confidence breakdown before anything is changed.
  2. One namespace at a time. Seed the mapping for that namespace, run the full flow, reconcile, then move to the next. The manifest is the running record across runs.
  3. Batch the review. The high-confidence tier clears in bulk; the team's reviewer only handles the medium and judgement tiers.
  4. Keep the audit. Every run adds to the tracking experiment, so program-level reporting (approval rate, override rate, failures) comes straight from the data.

To reset a trial between runs, re-run Stage 2 (it truncates and repopulates the manifest). Patched notebooks can be restored from the Volume backups or regenerated with the seed script.

Reference kinds and scoring

Confidence is rule-based and explainable, never a black-box model. Every score maps to a reason a reviewer can read. The reference kinds the scanner emits, and how each scores:

Reference kindExampleTypical score
sql_table_refSELECT ... FROM risk_db.exposures0.95 if exact mapping hit
py_table_refspark.read.table("sales_db.transactions")0.95 if exact mapping hit
mount_path_literalspark.read.parquet("/mnt/sales-raw/2024")0.90 if mount resolves
dynamic_mountf"/mnt/{name}/daily"0.70 (registry non-empty)
dynamic_tablespark.table(f"{db}.gl_accounts")0.30 (built at runtime)
anyno mapping hit / unknown namespace0.20

Proposed replacements: table refs become uc_catalog.uc_schema.uc_table; mount literals become a UC Volumes path that preserves the sub-path; dynamic references carry no proposal and wait for the reviewer's target.

Review routing

≥ 0.85Highreview_simple: one UPDATE approves the batch

Tune the thresholds and per-rule scores in scripts/scanner/core.py (propose and _tier). Keep every rule explainable: each should map to a one-sentence justification.

Reconciliation layers

Stage 5 compares each migrated table's source and UC target. What each layer checks:

LayerQuestion it answersOn mismatch
Row countDo source and target have the same number of rows?Manifest rows for that table set to failed
Column setAre the same columns present on both sides?Manifest rows for that table set to failed
Decision logWho approved what, at what confidence, when?One tracking run per decision, plus a summary
ResultLikely cause
Row count mismatchThe target table was not fully loaded, or a filter differs. This is a data-migration gap, not a code-patch gap.
Column missing on targetTarget schema differs from source. Confirm the table migration completed.
Reference flagged but table looks fineThe patched reference points at the wrong target. Re-open the row in review and override.

Troubleshooting

SDK auth fails with "match ... use --profile to specify"

This happens when two CLI profiles share the same workspace host. Select the profile with the environment variable rather than the SDK keyword: DATABRICKS_CONFIG_PROFILE=<profile> python3 scripts/jobs/stage1_discovery.py .... The stage scripts already do this when you pass --profile.

Discovery finds no mounts

On serverless, dbutils.fs.mounts() is unavailable, so the job uses a configured mount set and logs the fallback. Seed mount_point_registry from your known mounts, or configure the fallback set in scripts/jobs/stage1_discovery.py.

Apply cannot write the backup

The backup Volume must exist. The apply job creates it (CREATE VOLUME IF NOT EXISTS <catalog>.<schema>.backups); if that fails, check the run-as identity has CREATE VOLUME and write rights on the schema.

No external network in the workspace

The whole flow runs in-workspace with no external egress. The only external need is the Python packages (sqlglot, databricks-sdk, mlflow, ipywidgets); install them from your internal package proxy or pre-bundled wheels. No stage makes an outbound call.

A reference was patched incorrectly

Restore the asset from its backup_path in the Volume, re-open the manifest row in the review surface, override it with the correct target, and re-run Apply for that row.

FAQ

Does this move my data or migrate my tables?

No. It patches the code and dashboards that reference tables. Migrating the tables, grants, and metastore objects is the other half of UC migration, handled by your table-migration tooling. This toolkit complements it.

Is every change reversible?

Yes. Apply backs up each original to a UC Volume before patching, and re-import is overwrite, so you can restore any asset from its backup.

What about dashboards?

The scanner and manifest model dashboard queries as a reference kind, and the discovery step inventories dashboards. Notebook references are patched today; dashboard-query patching is the same pattern (extract query, scan, propose, apply) and extends from the same code.

Can it run with no internet access?

Yes. Everything runs inside the workspace. The only outside dependency is the Python packages, which you install from an internal proxy or bundled wheels.

How do I trust the confidence scores?

They are deterministic rules, not a model. Every score maps to a stated reason (see Reference kinds and scoring), and nothing below 0.85 is applied without a human approving it.
